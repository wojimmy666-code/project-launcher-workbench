本地项目执行管理台方案

把散落在不同硬盘位置的项目、页面、.exe、.bat、命令行工具、文档和文件夹集中到一个清晰的表格运维型工作台里。

浅色界面 表格运维型 本地运行 JSON 配置 白名单启动

1. 推荐形态

采用“本地 Web 工作台 + 本地后端 + JSON 配置”的方式。双击启动脚本后，自动打开本机页面。

浏览器页面
  ↓
本地后端服务
  ↓
projects.json 配置文件
  ↓
启动 exe / bat / npm / python / 打开网页 / 打开目录
双击 start-workbench.bat
  ↓
启动本地后端服务
  ↓
自动打开 http://localhost:3344

2. 页面设计

界面使用 02 表格运维型：左侧分类，中间高密度表格，顶部搜索和批量操作。

左侧分类栏全部项目、运行中、前端页面、后端服务、批处理脚本、命令行工具、文档和收藏。
顶部工具栏搜索框、状态筛选、类型筛选、批量检查、批量启动、新增项目。
项目表格名称、状态、类型、路径、端口、网址、最近启动和操作按钮。

项目行操作

  • 启动、停止、重启。
  • 打开网页、打开目录。
  • 查看日志、编辑配置。

3. 项目类型

类型用途示例
exe启动软件D:\Tools\App\app.exe
bat执行批处理D:\Scripts\start.bat
cmd执行命令npm run dev
url打开网页http://localhost:3000
folder打开目录D:\Projects\xxx
file打开文件E:\Docs\说明.docx

4. 配置文件设计

所有项目放在 projects.json，页面不写死项目。后续新增项目只改配置。

5. 状态判断

状态尽量基于真实检测，不只看是否点过启动。

运行中

端口可访问，或进程存在。

未启动

端口不通，进程不存在。

异常

启动过但端口未通，或命令异常退出。

未知

没有配置端口或进程检测方式。

6. 后端接口设计

接口方法用途
/api/projectsGET获取项目列表
/api/projects/:id/statusGET获取单个项目状态
/api/status/allGET批量获取状态
/api/projects/:id/startPOST启动项目
/api/projects/:id/stopPOST停止项目
/api/projects/:id/restartPOST重启项目
/api/projects/:id/open-urlPOST打开项目网址
/api/projects/:id/open-folderPOST打开项目目录
/api/projects/:id/logsGET查看日志

7. 安全策略

  • 只能启动 projects.json 中登记过的项目。
  • 页面不能提交任意 shell 命令。
  • 启动前检查路径是否存在。
  • .bat、.exe、cmd 类型分开处理。
  • 高风险命令需要在配置里显式标记。
  • 默认仅监听 127.0.0.1，不开放局域网访问。
  • 日志中不要记录密码、token、密钥。
{
  "server": {
    "host": "127.0.0.1",
    "port": 3344
  },
  "security": {
    "allowOnlyConfiguredProjects": true,
    "confirmDangerousActions": true,
    "allowNetworkAccess": false
  }
}

9. 开发阶段

阶段 1：静态页面定稿确定表格、筛选、状态标签、日志弹窗和配置详情弹窗。
阶段 2：配置驱动从 projects.json 读取项目，搜索和筛选基于配置字段。
阶段 3：真实启动能力后端启动 exe、bat、固定命令，打开网页和目录。
阶段 4：状态和日志增加端口检测、进程检测、最近启动、异常状态、日志查看。
阶段 5：增强功能批量启动组合、收藏、配置编辑、备份配置、桌面封装。

10. 最终建议
